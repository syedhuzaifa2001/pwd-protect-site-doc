Security Implementation Brief

How the login & content protection works

A plain-English and technical explanation of the access control protecting the QFZ Observatory file — what method is used, what is stored, and what it does and does not defend against.

Document type: Technical brief Scope: Single-file (offline) HTML app Classification: Shareable
1

Summary in one paragraph

The QFZ Observatory is a single, self-contained HTML file. Its entire content is encrypted and only unlocks when a user enters the correct username and password on a login screen. Crucially, the password is never stored anywhere in the file — not as plain text, not encrypted, not hashed. The password is used only as the key that unscrambles the content at the moment of login. Because nothing secret is written into the file, there is nothing to "extract" or "decrypt back" into a password — the only conceivable attack is guessing the password, which is deliberately made slow and expensive.

2

The core principle

This is the single most important idea, and the one most often misunderstood:

The password is a key, not a stored value.
We do not store your password and compare what users type against it. Instead, the password is fed through a one-way process to produce an encryption key, and that key is used to scramble the app content during setup. At login, the entered password regenerates the key and attempts to unscramble the content. Right password → it works. Wrong password → mathematically invalid output, access denied.

An analogy: think of the content as a meal cooked with a secret spice blend (your password). We save the cooked meal in the file, but we throw the spice blend away. Nobody can taste the meal and reconstruct the blend — they can only un-cook it by supplying the exact same blend again.

3

How it works, step by step

There are two distinct phases — a one-time setup that produces the file, and the unlock that happens every time someone signs in.

Phase A — Setup

Performed once, when the protected file is built.
1. Generate a random saltA random value created fresh for this file. Public, not secret.
2. Derive the keyPassword + salt run through PBKDF2 (250,000 rounds) → a 256-bit key.
3. Encrypt the appThe whole app is encrypted with AES-256-GCM using that key.
4. Save & discardSalt, IV and ciphertext are written to the file. The password and key are discarded.

Phase B — Unlock

Performed every time a user signs in.
1. User enters credentialsUsername and password typed into the login screen.
2. Re-derive the keySame PBKDF2 process rebuilds the key from password + the stored salt.
3. Attempt decryptionAES-GCM tries to decrypt. Its built-in integrity tag verifies correctness.
4. Show or rejectValid → the app is revealed. Invalid → "incorrect password", nothing exposed.
4

Cryptographic methods used

All algorithms are industry-standard and run through the browser's built-in Web Crypto API (crypto.subtle) — no custom or homemade cryptography is used.

PurposeMethodDetails
Content encryptionAES-256-GCMAuthenticated symmetric encryption, 256-bit key — the strongest standard tier. GCM also provides tamper detection.
Key derivationPBKDF2HMAC-SHA-256, 250,000 iterations. Stretches the password into the key slowly, to make password guessing expensive.
Salt16 random bytesUnique per file. Prevents pre-computed ("rainbow table") attacks. Stored in the file; not secret.
IV / nonce12 random bytesInitialisation vector for AES-GCM. Stored in the file; not secret.
Username bindingGCM additional
authenticated data		The username is bound into the encryption, so both username and password must be correct to unlock — and neither is stored.

What the file actually contains

Anyone can open the file in a text editor or browser console. Here is everything access-related they would find — and notice what is not there:

saltrandom bytesPUBLIC · safe
ivrandom bytesPUBLIC · safe
ciphertextthe encrypted app — useless without the keyPUBLIC · safe
password— not present —NEVER STORED
derived key— not present —NEVER STORED

The salt, IV and ciphertext are safe to be public because they are meaningless without the password. There is no stored secret to read, reverse, or "decrypt out" of the file.

6

Login & session behaviour

  • A branded login screen requests a username and password before any content is shown.
  • On a correct unlock, the session stays open while the browser is open (it persists across page reloads) and is cleared on Sign out or when the browser/tab is closed.
  • During an active session, the derived key (not the password) is held in the browser's per-tab session storage so reloads do not require re-typing. This only ever exists after a successful login and is discarded on sign-out.
  • The unlocked app is rendered using the page's own origin (via an inline document), so its internal navigation works identically whether the file is opened from a web server or directly from disk.
7

What it protects against — and what it doesn't

Protected against

  • Reading the password from the file or browser console — it isn't there.
  • "Decrypting" or reversing the password from the salt — mathematically impossible; the salt holds no password information.
  • Viewing the content without logging in — it exists only as encrypted bytes until unlocked.
  • Pre-computed / rainbow-table attacks — defeated by the random salt.
  • Silent tampering with the file — AES-GCM detects modification.

Residual risks

  • Guessing the password. The only real attack: try many passwords until one decrypts. PBKDF2's 250k iterations make this slow, but a weak or predictable password can still be guessed.
  • This is a self-contained file, so access cannot be revoked remotely once shared. Changing the password requires re-issuing the file.
  • Anyone who is already logged in can see the content (as intended).
The password is the entire lock.
Since guessing is the only viable attack, the protection is exactly as strong as the password is hard to guess. Avoid passwords based on the username, dates, or common words. Use a long, random password and store it in a password manager.
8

When to use this — and when to go further

This approach is the strongest protection achievable for a single offline file with no server. It is well suited to gating a self-contained demo or document behind a shared credential.

Consider a backend if you need:
per-user accounts, the ability to revoke access after sharing, audit logs of who opened it, password resets, or rate-limiting of guesses. Those require a server and cannot be delivered by a standalone file alone.
